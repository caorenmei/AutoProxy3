# AutoProxy3 设计文档

## 背景与目标

AutoProxy3 将实现一个基于 Go 的代理服务器，支持普通 HTTP 代理请求与 CONNECT 请求入站接入，并根据在线规则、本地规则、自动探测规则决定是直连目标地址还是通过上游代理转发。系统同时提供本地管理接口、命令行重载入口、结构化日志、测试与 CI/CD。

本次设计目标是：

1. 在单进程模型下完成完整功能交付，避免过度拆分进程带来的复杂度。
2. 保证规则系统、代理转发、管理接口三者边界清晰，可独立测试。
3. 满足 `SPEC.md` 与 `AGENTS.md` 对文档、测试、工作流和目录结构的要求。

## 约束与澄清结论

以下结论已在设计确认阶段明确：

1. 入站请求必须兼容普通 HTTP 代理请求与 CONNECT 请求；若使用上游代理，则统一通过 CONNECT 建立出站隧道。
2. 本地规则与自动探测规则仅按目标主机名或域名匹配，不匹配 URL 路径。
3. 在线 AutoProxy 规则中的 `@@` 排除规则具备全局最高优先级；即使本地规则或自动探测规则命中，也必须最终直连。
4. 自动探测只在未命中任何规则、且直连 TCP 建链失败时触发；HTTP 4xx/5xx、应用层错误或其他非 TCP 建链失败都不触发。
5. 命令行 `reload_web_rules`、`reload_custom_rules`、`reload_rules` 作为本地管理客户端实现，通过请求管理接口控制运行中的代理实例。
6. `reload_custom_rules` 需要同时重载本地规则与自动探测规则，因为自动探测规则文件可能被外部修改。
7. 若 `reload_custom_rules` 的文件重载过程失败，继续使用当前内存快照并记录明确错误日志。
8. 当规则判定应走上游代理但未配置 `upstream_proxy` 时，请求回退为直连，并记录明确告警日志，而不是阻止服务启动。
9. 在线规则刷新失败时，保留当前内存规则与本地缓存，不清空已生效规则。
10. CI 默认使用当前稳定版 Go。

## 总体架构

项目采用单二进制、分层模块化设计。程序整体由配置层、规则层、代理层、管理层、日志层与 CLI 层组成，各模块职责如下：

### `src/cmd/autoproxy3`

程序入口，负责解析命令行参数并分发执行路径：

1. 正常启动代理服务。
2. 输出版本信息。
3. 输出帮助信息。
4. 作为本地管理客户端调用 `reload_*` 接口。

### `src/internal/config`

负责读取 JSON 配置文件并完成默认值填充、路径归一化和配置校验。该模块统一处理：

1. 监听地址。
2. 上游代理地址。
3. 在线规则、本地规则、自动探测、管理接口、日志配置。
4. 相对路径基于配置文件所在目录解析。

### `src/internal/rules`

规则系统提供线程安全的统一查询接口，内部拆分为：

1. `webrules`：下载、解码并解析 Base64 编码的 AutoProxy 规则。
2. `customrules`：解析本地纯文本通配规则。
3. `autodetect`：维护自动探测规则文件与内存快照。
4. `matcher`：统一进行规则优先级判定。

规则优先级固定为：

1. 在线规则的 `@@` 排除规则。
2. 在线正向规则、本地规则、自动探测规则的任意正向命中。
3. 未命中任何规则时默认直连。

### `src/internal/proxy`

代理主服务，负责：

1. 接收入站普通 HTTP 代理请求和 CONNECT 请求。
2. 根据规则引擎决策执行直连或上游代理转发。
3. 在符合条件时触发自动探测。
4. 记录请求日志和错误日志。

### `src/internal/management`

管理 HTTP 服务只监听 `127.0.0.1:<listen_port>`，提供：

1. `GET /`：返回版本号、已启用功能与规则状态摘要。
2. `POST /reload_web_rules`：重载在线规则。
3. `POST /reload_custom_rules`：同时重载本地规则与自动探测规则。
4. `POST /reload_rules`：重载全部规则。

### `src/internal/logging`

基于 `slog` 统一初始化日志系统，支持文本与 JSON 两种输出格式，并通过滚动文件写入器支持日志轮转。

### `src/internal/buildinfo`

集中维护版本号与已启用功能信息，供 CLI 输出和管理首页复用。

## 核心运行流程

### 请求判路流程

每个请求会先抽取目标主机与端口，再经过统一决策流程：

1. 查询是否命中在线规则中的 `@@` 排除规则；若命中，强制直连。
2. 否则查询在线正向规则、本地规则、自动探测规则是否任意命中；若命中，则应走上游代理。
3. 若未命中任何规则，则先尝试直连。
4. 若决策结果为走上游代理，但未配置 `upstream_proxy`，则降级为直连并记录 `warn` 日志。

### 转发执行流程

1. 入站普通 HTTP 代理请求：代理解析目标站点，在直连或上游隧道内发起出站 HTTP 请求，并返回响应。
2. 入站 CONNECT 请求：代理建立到目标站点的 TCP 隧道，随后进行双向字节转发。
3. 通过上游代理时，无论上层协议为何，统一先对上游代理发送 CONNECT 请求建立隧道。

### 自动探测流程

自动探测只在以下条件全部满足时触发：

1. 当前目标未命中任何规则。
2. 直连 TCP 建链失败。
3. 失败次数达到 `max_attempts`。
4. 已配置 `upstream_proxy`。

在满足上述条件后：

1. 改为通过上游代理再次尝试。
2. 若建立连接并完成转发成功，则将目标主机写入自动探测规则文件。
3. 写入成功后立即刷新内存快照，使后续请求可直接命中自动探测规则。

自动探测不会因为 HTTP 状态码异常、TLS 之后的应用层失败或其他非 TCP 建链失败而触发。

## 规则更新与状态管理

系统维护单一 `RuleEngine` 作为规则状态入口。请求路径只读取内存快照，更新路径通过受控方法原子替换状态。

### 在线规则

1. 支持启动时下载与定时刷新。
2. 更新流程为：HTTP 下载 → Base64 解码 → AutoProxy 解析 → 构建新快照 → 原子替换内存状态 → 写入本地缓存。
3. 任一环节失败都不覆盖当前内存状态和已有缓存，只记录错误日志。

### 本地规则

1. 从文本文件读取规则。
2. 忽略空行与 `#` 注释。
3. 管理接口触发 `reload_custom_rules` 或 `reload_rules` 时重新构建快照并原子替换。

### 自动探测规则

1. 规则文件格式与本地规则一致。
2. 记录主机名时需要去重。
3. 写盘成功后立即刷新对应快照。
4. 若写盘失败，请求结果保持成功返回，但需记录明确错误日志。
5. 管理接口触发 `reload_custom_rules` 或 `reload_rules` 时，也需要从自动探测规则文件重新读取并构建快照；若重载失败，则保留当前内存快照并记录错误日志。

### 并发安全

1. 读路径使用只读快照，避免请求处理期间长期持锁。
2. 写路径通过写锁或原子指针替换保证视图一致。
3. 后台定时刷新任务支持优雅停止，避免 goroutine 泄漏。

## 管理接口与 CLI 关系

命令行中的 `reload_web_rules`、`reload_custom_rules`、`reload_rules` 不直接修改规则文件，而是作为本地 HTTP 客户端调用管理接口。

这样设计的好处是：

1. 所有运行时规则更新入口统一复用管理层逻辑。
2. 可以保证重载操作作用于真实运行中的代理实例。
3. CLI 行为更容易通过集成测试覆盖。

其中 `reload_custom_rules` 的语义固定为：同一次请求内同时尝试重载本地规则和自动探测规则；任一子步骤失败时，不替换对应内存快照，并在响应体与日志中明确给出失败信息。

管理接口响应应返回结构化 JSON，明确区分成功、部分成功和失败原因。

## 错误处理与日志

日志系统需要满足以下要求：

1. 使用 `slog`。
2. 支持 `debug`、`info`、`warn`、`error`。
3. 支持 `text` 与 `json` 两种输出格式。
4. 支持标准输出与文件输出。
5. 支持基于文件大小与备份数量的日志轮转。

错误处理原则如下：

1. 配置文件读取或校验失败时，启动失败并返回明确错误。
2. 管理接口重载失败时，返回非成功状态码和清晰错误体。
3. 在线规则刷新失败时保留旧状态，不做静默降级。
4. 规则要求使用上游但未配置上游时，保留服务可用性，按直连处理并输出告警日志。
5. 自动探测写盘失败时，不吞掉错误日志，但不回滚已成功的转发结果。

## 测试策略

本项目采用 TDD，先写测试再写实现。测试分为单元测试与集成测试两层。

### 单元测试

单元测试目标是覆盖纯逻辑模块并满足规格要求的 100% 覆盖率，重点包括：

1. 配置解析、默认值与路径归一化。
2. AutoProxy 规则解析和 `@@` 优先级。
3. 本地规则与自动探测规则的通配匹配。
4. 规则合并后的判定结果。
5. 自动探测触发条件。
6. 管理客户端请求构造。
7. 日志配置初始化。

### 集成测试

集成测试通过 `httptest`、本地 TCP 监听器和伪上游代理覆盖真实行为，重点包括：

1. 直连普通 HTTP 代理请求。
2. 直连 CONNECT 隧道请求。
3. 通过上游 CONNECT 转发普通 HTTP 与 CONNECT 请求。
4. 在线规则刷新与应用。
5. `reload_custom_rules` 同时重载本地规则和自动探测规则，并验证失败时保留原内存快照。
6. 自动探测触发、写盘与后续命中。
7. 管理接口与 CLI `reload_*` 的联动。

## 实现建议顺序

推荐实现顺序如下：

1. 初始化 Go 模块、基础目录和公共构件。
2. 先完成配置与规则解析相关测试和实现。
3. 再完成规则引擎与自动探测逻辑测试和实现。
4. 随后完成代理转发核心流程。
5. 再接入管理接口与 CLI 管理客户端。
6. 最后补齐日志、文档、配置样例、CI/CD 与 README。

这个顺序能保证核心逻辑先稳定，再逐层接入对外接口和交付物。
