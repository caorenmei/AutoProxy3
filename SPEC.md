# SPEC.md - 项目规格说明

本项目开发一个 HTTP 代理服务器，具备以下功能：

- **请求转发**：将客户端的 HTTP 请求转发到目标服务器，并将响应返回给客户端。
- **上游代理**：能够配置上游代理服务器。
- **本地规则**：支持用户定义的规则列表，如果配置了上游代理，可以根据规则决定是否使用上游代理。
- **在线规则**：支持在线的AutoProxy规则列表，能够定期更新。
- **自动探测**：配置了此选项，当目标地址不匹配任何规则时，建立 TCP 连接失败重试指定次数后，使用上游代理代理成功创建连接且转发请求成功后，将目标地址记录到自动探测的规则列表中，以便后续请求直接使用上游代理。
- **规则重载**：提供接口允许用户手动重载规则列表。
- **日志记录**：记录代理服务器的运行日志，包括请求日志、错误日志等。

## 配置结构

使用 JSON 格式的配置文件。支持配置上游代理、在线规则列表、本地规则列表、自动探测功能、以及管理接口的相关设置。

当请求匹配在线规则列表、本地规则列表、自动探测列表任意一项时，代理服务器将使用上游代理转发请求，否则直接转发到目标服务器。在线规则列表中的 `@@` 排除规则具备全局最高优先级，命中时强制直连。

在线规则列表的格式为 AutoProxy 格式，HTTP 响应体编码为 BASE64，解码后为纯文本格式。在线规则列表将定期更新，更新间隔可配置。示例规则：

```
[AutoProxy 0.2.9]

|http://specific-site.com/forbidden
||twitter.com
||youtube.com

@@||apple.com
@@||icloud.com

```

本地规则列表与自动探测规则列表的格式为纯文本格式，支持通配符，使用换行分割规则，使用 # 进行注释。规则仅按目标主机名或域名匹配，不匹配 URL 路径。示例规则：

```
# Google 的请求将使用上游代理转发
*.google.com

# Facebook 的请求将使用上游代理转发
*.facebook.com

```

自动探测功能将记录探测结果到自动探测的规则列表中，规则列表的格式与本地规则列表相同。

管理接口提供 HTTP 接口允许用户重载规则列表。管理接口的监听地址为 localhost，仅允许本地访问。
管理 HTTP 服务器将提供以下接口：
- `GET /`: 输出当前版本号和已开启的功能。
- `POST /reload_web_rules`：重载在线规则列表。下载或解析失败时保留当前内存快照与本地缓存，不覆盖已生效规则。
- `POST /reload_custom_rules`：同时重载本地规则列表与自动探测规则列表。仅当两者均解析成功时才替换内存快照，任一失败时保持原有快照不变。
- `POST /reload_rules`：重载所有规则列表（在线规则列表、本地规则列表和自动探测规则列表）。任一子步骤失败时，对应快照保持原有状态不变。

日志配置设置项允许用户配置日志的输出级别、输出格式、以及日志文件的路径等。日志级别包括 debug、info、warn、error 等，用户可以根据需要选择合适的日志级别来记录代理服务器的运行状态和错误信息。
使用 slog 进行日志记录，支持文本格式和 JSON 格式两种日志输出格式。

等价的 YAML 配置示例如下（使用 YAML 格式是为了更直观地展示配置结构，但是程序实际使用的是 JSON 格式）：

```yaml

# 代理服务器的监听地址，默认为"localhost:1080"。
listen_addr: "localhost:1080"
# 上游代理，可选。如果配置了上游代理，代理服务器将根据规则列表决定是否使用上游代理转发请求。
upstream_proxy: "http://upstream-proxy.example.com:8080"
# 在线规则列表配置，可选，如果配置了在线规则列表，代理服务器将定期从指定URL下载规则列表并更新。
web_rules:
  # 是否启用在线规则列表功能，默认为true。
  enabled: true
  # 在线规则列表的URL，HTTP 响应体编码为 BASE64，解码后为 AutoProxy 格式。
  url: "http://rules.example.com/rules.txt"
  # 本地缓存规则列表的路径，推荐使用相对路径
  cache_path: "web_rules.txt"
  # 规则列表的刷新间隔，单位为秒，可选。不设置或设置为0表示不自动刷新。
  refresh_interval: 3600
  # 启动时是否立即下载规则列表，默认为true。
  download_on_start: true
# 本地规则列表，可选。如果配置了本地规则列表，代理服务器将根据规则列表决定是否使用上游代理转发请求。
custom_rules:
  # 是否启用本地规则列表功能，默认为true。
  enabled: true
  # 本地规则列表的路径，推荐使用相对路径
  path: "custom_rules.txt"
# 自动探测功能配置，可选。如果启用自动探测功能，当请求失败时，代理服务器将自动探测是否需要使用上游代理。
auto_detect:
    # 是否启用自动探测功能，默认为false。
    enabled: true
    # 探测次数。对同一目标主机的直连 TCP 连接失败进行累计计数，当累计失败次数达到此值时，认为需要使用上游代理。默认为3次。使用上游代理成功创建连接且转发请求成功后，将目标地址记录到自动探测的规则列表中，以便后续请求直接使用上游代理。
    max_attempts: 3
    # 自动探测的规则列表路径，推荐使用相对路径。
    rules_path: "auto_detect_rules.txt"
# 管理接口配置，可选。如果配置了管理接口，代理服务器将提供HTTP接口允许用户重载规则列表。
management:
    # 是否启用管理接口，默认为true。
    enabled: true
    # 管理接口的监听端口，默认为9091。监听地址为localhost，仅允许本地访问。
    listen_port: 9091
# 日志配置设置项允许用户配置日志的输出级别、输出格式、以及日志文件的路径等。
logging:
    # 日志级别，包括 debug、info、warn、error
    level: "info"
    # 日志输出格式，包括 text、json
    format: "text"
    # 日志文件的路径，如果不设置或设置为null表示不输出到文件，日志将输出到标准输出。
    file_path: "logs/proxy.log"
    # 日志文件的最大大小，单位为MB，默认为10MB。当日志文件达到最大大小时，将自动轮转日志文件。
    max_size: 10
    # 日志文件的最大保留数量，默认为5。当日志文件达到最大保留数量时，将删除最旧的日志文件。
    max_backups: 5
```

## 命令行参数

支持以下命令行参数：
- `config`：指定启动的配置文件路径，默认为`config.json`。
- `version`：显示版本信息并退出。
- `help`：显示帮助信息并退出。
- `reload_web_rules`：重载在线规则列表。
- `reload_custom_rules`：同时重载本地规则列表与自动探测规则列表。
- `reload_rules`：重载所有规则列表（在线规则列表、本地规则列表和自动探测规则列表）。

## 技术规格

- **编程语言**：Golang
- **实现原理**：入站请求兼容普通 HTTP 代理请求与 CONNECT 请求；通过上游代理转发时，统一使用 **HTTP CONNECT** 方法建立出站隧道。

## 运行时行为

### 上游代理未配置时的回退

当规则判定应使用上游代理转发，但配置文件中未设置 `upstream_proxy` 时，请求将回退为直连目标服务器，并输出 `warn` 级别告警日志。此行为不会阻止代理服务启动。

### 在线规则下载与缓存

在线规则下载请求本身复用同一套规则判路语义：
- 首次启动且本地尚无在线规则缓存时，由于系统无法提前知道规则 URL 的判路结果，首次下载默认直连。
- 一旦本地已有缓存或当前进程已加载过在线规则快照，后续下载与刷新都必须先根据当前规则集判定该 URL 是否应走上游；命中代理且已配置上游时，经上游代理执行下载。

启动阶段在线规则加载行为：
- 若 `web_rules.enabled=false`，跳过在线规则加载。
- 若 `download_on_start=true`，优先尝试从远程 URL 下载并加载。
- 若下载失败且本地缓存文件存在，则回退加载本地缓存，继续启动。
- 若下载失败且本地缓存也不存在，则启动失败。

运行期间定时刷新失败时，保留当前内存中的在线规则快照和本地缓存，不覆盖已生效规则，仅记录错误日志。

### 自动探测触发条件

自动探测仅在以下全部条件满足时触发：
1. 当前目标主机未命中任何规则（包括在线规则的正向规则、本地规则、自动探测规则，但不包括 `@@` 排除规则）。
2. 直连 TCP 连接建立失败（HTTP 状态码异常、TLS 握手失败、应用层错误等不触发自动探测）。
3. 对同一目标主机的直连 TCP 建链累计失败次数达到 `auto_detect.max_attempts`。
4. 已配置 `upstream_proxy`。

触发后，改为通过上游代理再次尝试。若建立连接并完成转发成功，则将目标主机追加写入自动探测规则文件，去重后刷新内存中的自动探测规则快照。若写盘失败，请求结果保持成功返回，但记录错误日志。

## 测试驱动开发

在开发过程中，将采用测试驱动开发（TDD）的方式，先编写测试用例，再实现功能代码。测试用例将覆盖代理服务器的所有功能，包括请求转发、规则匹配、自动探测等。
测试用例将使用 Go 的内置测试框架编写，并且会模拟客户端请求和目标服务器响应，以验证代理服务器的行为是否符合预期。

### 单元测试

单元测试的覆盖率需要达到100%，确保每个功能点都经过充分测试。

### 集成测试

集成测试将模拟真实的使用场景，验证代理服务器在实际环境中的表现。集成测试将包括以下场景：
- 使用上游代理转发请求。
- 不使用上游代理直接转发请求。
- 在线规则列表的更新和应用。
- 本地规则列表的更新和应用。
- 自动探测功能的触发和规则列表的更新。
- 管理接口的使用，包括手动重载规则列表。

### Github CI/CD

将配置 Github Actions 进行持续集成和持续部署（CI/CD）。每次提交代码后，CI 将自动运行所有测试用例，确保代码的质量和稳定性。只有当所有测试用例通过后，才允许合并代码到主分支。
