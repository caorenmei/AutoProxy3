# AGENTS.md — 核心指令与工作规范

本文件定义了 Code Agent 在本项目中的最高行为准则。  
Agent 在执行任何任务前，**必须**阅读并严格遵守以下全部规定。

---

## 🧩 1. 工作流

### 1.1 Superpowers 工作流
- 所有任务**必须**基于 Superpowers 工作流执行，并遵循其阶段划分与质量标准。

### 1.2 文档同步（三向对齐）
- 进行逻辑变更、架构调整或新增功能时，**必须同步更新**以下三类文档，确保它们相互一致：
  - **用户文档**：`docs/` 目录下的架构、API 定义、开发指南、业务流程等。
  - **Superpowers 文档**：`docs/superpowers/` 下的工作流定义与流程说明。
  - **代码文档**：源代码中的注释、JSDoc/Docstring 等。
- 目标：用户文档、Superpowers 文档与代码文档始终保持 100% 一致。
- **变更日志强制记录**：每次完成上述同步后，必须在 `docs/changelog/` 下创建对应日志文件，文件名格式为 `YYYYMMDD-NNN-brief-change-description.md`（NNN 为当日三位数字序号，从 001 开始递增；`brief-change-description` 需使用英文小写字母与连字符）。日志须使用简体中文，明确列出涉及的用户文档、Superpowers 文档和代码文档路径，并确认三项全部已对齐。该日志文件将作为三向对齐的唯一记录，必须与代码变更一同提交。

### 1.3 分支与集成流程
- **开发环境**：使用 Git Worktree 进行特性开发。
- **代码提交**：所有变更通过 Pull Request 提交。
- **合并条件**：密切关注 Pull Request 触发的 GitHub Actions CI/CD 流程，**仅当所有测试通过后**方可合并到主分支。

## 🧠 2. 开发行为准则

以下准则旨在减少常见错误，提高代码质量与协作效率。

### 2.1 简单至上
- **只编写解决问题的最小代码量**：
  - 不实现未明确要求的功能。
  - 不为单一用途的代码引入抽象层。
  - 不添加未被要求的“灵活性”或“可配置性”。
  - 不为不可能发生的场景编写错误处理逻辑。
- **代码自审**：若一段实现超过了必要的复杂度（例如本应 50 行却写了 200 行），必须重写。永远反问：“一个资深工程师是否会认为这段代码过度复杂？” 若答案为是，立即简化。

### 2.2 精准修改
- **只触碰必须改的代码**：
  - 禁止“顺便改进”相邻的代码、注释或格式。
  - 禁止重构未涉及的功能。
  - 严格遵循已有代码风格，即使个人偏好不同。
  - 若发现无关的死代码或问题，只能提醒，**禁止擅自删除**。
- **清理自身引入的孤立代码**：
  - 因你的修改而变得无用的导入、变量、函数必须删除。
  - 但本次修改前就已存在的死代码，除非明确要求否则不得触碰。
- **测试标准**：每一行改动都必须能直接追溯到用户的具体请求。

### 2.3 重构准则
重构仅允许在以下受限情况下进行，且必须遵循严格的边界：

- **允许重构的情形**：
  - 当前任务的实现若不强依赖于某处重构，则会导致代码逻辑混乱或产生明显缺陷。
  - 重构是完成用户明确要求的任务中不可避免的一环（例如需要提取共用方法以消除两处重复逻辑）。
  - 用户明确要求进行特定重构。
- **禁止重构的情形**：
  - 单纯出于“改善代码风格”、“提升可读性”或“符合个人偏好”而进行的非必要改动。
  - 改动范围超出当前任务直接关联的模块或函数。
  - 以重构为名引入新的抽象层或设计模式。
- **重构前必须满足的条件**：
  - 相关代码已有可运行的测试保护，或先补足测试使重构有安全网。
  - 必须向用户说明重构的必要性、范围和预期收益，并获得明确确认。
  - 重构与功能修改必须分步提交，禁止混合在同一 commit 中。

## 🇨🇳 3. 语言规范

- **对话与规划**  
  所有与用户的交互、任务规划、总结、思考过程**一律使用简体中文**。

- **文档与注释**  
  生成的 Markdown、JSDoc、Docstring 等必须使用**简体中文**。

- **代码与字符串**  
  新增或修改的代码、测试代码、配置键名、命令示例、字符串字面量（含测试断言消息与运行时字符串）**默认使用英文**；未触动的存量代码保留原状。

- **技术术语**  
  文档或注释中的技术缩写、协议名及约定俗成的术语可以保留原文。

---

## 📝 4. 注释与文档要求

- **公共成员强制注释**  
  所有 Public API、Exported Class、Public Method 必须包含详尽的简体中文注释。

- **注释内容**  
  需涵盖功能描述、参数说明（类型/含义）、返回值定义以及可能抛出的异常。

## 💬 5. 交互准则

- **强制询问**  
  任何时候遇到不确定因素，**必须使用 `AskUser` 或 `AskUserQuestion` 工具**向用户确认。

- **主动推进**  
  在完成一轮任务后，**必须使用 `AskUser` 或 `AskUserQuestion` 工具**提出与当前上下文相关的后续问题或建议，引导项目持续迭代。

## 📂 6. 项目目录结构

项目基础目录遵循以下布局，以保证文档与代码的同步；具体子目录可根据语言与框架规范灵活扩展，但核心目录不可省略。

```text
.
├── .agents/
│   └── skills/               # Agent 技能定义
├── AGENTS.md                 # 本文件，Code Agent 最高准则
├── docs/
│   ├── architecture/         # 架构决策与核心逻辑
│   ├── api/                  # 接口定义与协议说明
│   ├── workflows/            # 开发、部署与业务流程
│   ├── guides/               # 开发规范与专项指南
│   └── superpowers/          # Superpowers 工作流文档
├── src/                      # 源代码（或依语言惯例命名，如 lib/、app/ 等）
└── README.md                 # 项目总览（简体中文）
```

> **扩展原则**：任何新增目录或文件，都必须在三向对齐时同步更新所有涉及路径引用的文档，并记录于变更日志。

---

> **提醒**：上述所有规则均为强制性要求。Code Agent 的任何行为若违反本文件，视为不符合项目规范的输出。
